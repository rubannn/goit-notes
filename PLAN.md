# План: превращение goit-notes в редактируемый веб-ресурс

## 1. Текущее состояние проекта

`goit-notes` — статический сайт-трекер домашних заданий GoIT:

- `index.html` — единственная страница, контейнер `#table-container`.
- `js/gentable.js` — вся логика: по `fetch` тянет `data/file-list.json`, затем каждый
  `data/dataXX.json`, рендерит таблицы, сворачивание/разворачивание блоков,
  всплывающие комментарии, подсчёт времени до дедлайна и сводку "срочные задачи".
- `data/*.json` — сами данные (курсы/блоки → строки-темы с `topic`, `link`,
  `lms_link`, `done`, `score`, `deadline`, `verified`, `comment{text, tasks,
  instructions, instructions_text, urls}`, `useful_links[]`, `optional`).
- `css/style.css` — оформление, анимации сворачивания.
- Данные редактируются **только вручную**, правкой JSON-файлов и коммитом в git.
  Все чекбоксы в UI `readOnly`. Сайт скорее всего отдаётся через GitHub Pages
  (репозиторий публичный, `origin` → `github.com/rubannn/goit-notes`, нет
  `package.json`/бэкенда).

Важный нюанс, который нашёлся по пути: `gentable.js` кладёт `rowData.topic` через
`innerHTML` (не `textContent`) в нескольких местах. Пока данные пишет только
владелец руками в JSON — это не проблема. Как только поле станет редактируемым
через форму в браузере, это открывает **stored XSS** и требует исправления (см. Фаза 6).

## 2. Выбранное решение

По итогам обсуждения с пользователем:

- **Хранение**: полноценный бэкенд + база данных (не GitHub API, не localStorage).
  JSON-файлы в `data/` перестают быть источником правды в рантайме — их заменяет БД.
- **Доступ**: сайт может оставаться публично читаемым, но редактирование доступно
  только владельцу — защищено паролем/токеном.

### Стек

| Слой      | Выбор                                                   |
|-----------|----------------------------------------------------------|
| Backend   | Cloudflare Pages Functions (serverless, по одной функции на маршрут вместо Express-роутера) |
| БД        | Cloudflare D1 — SQLite-совместимая БД, схема из раздела 3 переносится без изменений |
| Auth      | Один пароль владельца (секрет в Cloudflare, bcrypt-хэш) → сессионная httpOnly-cookie, сессии в D1 или Workers KV |
| Frontend  | Остаётся vanilla JS/HTML/CSS, `gentable.js` дорабатывается, framework не нужен |
| Хостинг   | Cloudflare Pages — деплой по git push из этого же репозитория, бесплатный тир, свой поддомен `*.pages.dev` или кастомный домен |

Хостинг выбран как ближайший аналог GitHub Pages: тот же workflow "push в
GitHub → автодеплой", не нужно администрировать сервер, бесплатный тариф.
Отличие от GitHub Pages — Cloudflare Pages умеет ещё и бэкенд (Functions + D1),
поэтому не нужно переезжать на отдельный хостинг под API.

**Альтернатива**: Render.com — Node.js/Express почти без переделки (ближе к
исходному варианту плана), тоже автодеплой по git push. Но на бесплатном
тарифе диск эфемерный — файл SQLite слетает при каждом перезапуске/редеплое,
если не платить за persistent disk. Имеет смысл, если важнее сохранить Express,
чем переписывать роуты под Functions.

Если удобнее на Python (в репозитории уже есть Python-домашки GoIT) —
Cloudflare Functions хуже поддерживает Python, поэтому для Python-варианта
логичнее выбрать альтернативу: Render + FastAPI + `sqlite3`/SQLModel.
Архитектура (схема, API, фазы) не меняется, меняется только исполнение бэкенда.

## 3. Модель данных

```
blocks                        rows
------                        ----
id            PK              id             PK
title                         block_id       FK -> blocks.id
order_num                     order_index
collapse      bool            topic
                               link
                               lms_link
                               done           bool
                               score          bool
                               deadline       text (dd.mm.yyyy)
                               verified       bool
                               optional       bool
                               comment        json  (text, tasks, instructions,
                                                      instructions_text, urls)
                               useful_links   json  ([{url, text}])
```

`comment` и `useful_links` остаются JSON-колонками (TEXT в SQLite) — это
редко изменяемые вложенные структуры, разворачивать их в отдельные таблицы
избыточно для объёма данных (~10 блоков × ~10 строк).

## 4. API

Публичное чтение остаётся открытым (как сейчас на GitHub Pages), мутации требуют сессии.

| Метод & путь                | Auth | Назначение |
|------------------------------|------|------------|
| `POST /api/auth/login`       | —    | пароль → httpOnly session cookie |
| `POST /api/auth/logout`      | —    | сброс сессии |
| `GET /api/blocks`            | нет  | все блоки со строками — заменяет связку `file-list.json` + `dataXX.json` |
| `POST /api/blocks`           | да   | создать блок (курс/модуль) |
| `PUT /api/blocks/:id`        | да   | title/order_num/collapse |
| `DELETE /api/blocks/:id`     | да   | удалить блок и его строки |
| `POST /api/blocks/:id/rows`  | да   | добавить строку-тему |
| `PUT /api/rows/:id`          | да   | изменить любое поле строки |
| `DELETE /api/rows/:id`       | да   | удалить строку |
| `PUT /api/blocks/reorder`    | да   | (опционально) массовое обновление order_num при drag&drop |

Мидлварь авторизации проверяет cookie/сессию перед всеми не-GET запросами.

## 5. Фронтенд

- `gentable.js` разделить на:
  - `api.js` — обёртки над `fetch('/api/...')`.
  - `render.js` — текущая логика рендера таблиц (почти без изменений).
  - `editor.js` — новая логика: кнопка "Войти", кнопка "Редактировать" на блоке/строке,
    инлайн-формы для полей строки, кнопки "Добавить тему"/"Добавить блок"/"Удалить".
- Экран логина: простая форма пароль → `POST /api/auth/login`, дальше все fetch с
  `credentials: 'include'`.
- Кнопки редактирования показываются только когда сессия активна (проверяется по
  ответу `GET /api/auth/me` при загрузке страницы).
- Все места, где сейчас `el.innerHTML = rowData.topic`, заменить на `textContent`,
  либо (если нужна разметка типа `<b>`) прогонять через белый список тегов
  (например, через `DOMPurify`) — без этого форма редактирования темы станет
  вектором XSS.

## 6. Миграция данных

Разовый скрипт `scripts/migrate-json-to-db.js`:
1. Прочитать `data/file-list.json`, затем каждый `data/dataXX.json`.
2. Создать таблицы `blocks`/`rows` в SQLite.
3. Вставить блоки в порядке из `file-list.json` (порядок сейчас определяет
   нумерацию `num` в заголовках), строки — в порядке `rows[]` внутри файла.
4. Сохранить исходные `data/*.json` в репозитории как архив/бэкап на момент миграции.

## 7. Бэкапы и история изменений

Раньше каждая правка данных была git-коммитом с историей — при переходе на БД это
свойство теряется. Компенсировать:
- Эндпоинт `GET /api/export` — отдаёт текущее состояние БД в том же формате
  `data/*.json` + `file-list.json`.
- Простой cron/скрипт (или кнопка в UI), который раз в день/неделю вызывает
  экспорт и коммитит снапшот в `data/` в git — история остаётся, просто не на
  каждую правку, а периодическими снимками.

## 8. Безопасность

- Пароль владельца — только bcrypt-хэш в переменной окружения, не в коде/репозитории.
- Сессионная cookie: `httpOnly`, `secure`, `sameSite=strict`.
- Rate-limit на `/api/auth/login` (защита от подбора пароля).
- Валидация входных данных на бэкенде (длины полей, формат `deadline`,
  URL-валидация для `link`/`lms_link`/`useful_links[].url`).
- Исправить XSS-риск с `innerHTML` (см. пункт 5) — обязательно перед тем как
  редактирование станет доступно из браузера.

- [ ] **Фаза 3** — авторизация владельца (логин/логаут, сессии, мидлварь).

- [ ] **Фаза 5** — UI редактирования: добавление/изменение/удаление темы и блока.
- [ ] **Фаза 6** — устранение `innerHTML`-XSS, санитизация ввода.
- [ ] **Фаза 7** — деплой на Render/Fly.io/Railway с persistent volume, секреты через env.

- Нужен ли markdown/рич-текст в `comment.text`, или простого текста достаточно.
- Нужно ли сохранить публичный read-only доступ для всех, или закрыть весь сайт
